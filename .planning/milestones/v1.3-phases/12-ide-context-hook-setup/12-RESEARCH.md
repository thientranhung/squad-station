# Phase 12: IDE Context & Hook Setup - Research

**Researched:** 2026-03-09
**Domain:** Rust file I/O, JSON merge, Markdown generation, Claude Code / Gemini CLI settings.json format
**Confidence:** HIGH

## Summary

Phase 12 adds two independent capabilities to the existing `squad-station` binary. First, `context.rs` is transformed from printing to stdout into a file-writing command that generates three Markdown workflow files under `.agent/workflows/` — these files give IDE orchestrators (Antigravity etc.) everything they need to delegate, monitor, and identify agents. Second, `init.rs` gains settings.json merge logic: when `.claude/settings.json` or `.gemini/settings.json` exists it parses the JSON, merges a hook entry, backs up the original, and writes the merged result; when the file does not exist it prints human-readable setup instructions to stdout.

The two capabilities are surgically independent. The `context.rs` rewrite does not touch DB logic, config loading, or agent registration — it only changes what happens with the fetched agent list. The `init.rs` settings.json logic is additive: existing init flow (DB register, tmux launch) runs first; hook setup is appended at the end.

No new Cargo dependencies are required. The standard library provides `std::fs` for file write/backup. `serde_json` (already a dependency) handles JSON parsing and merging. The `.agent/workflows/` directory creation uses `std::fs::create_dir_all` — an idempotent pattern already used in `config.rs` for DB directory creation.

**Primary recommendation:** Split Phase 12 into two plans: Plan 12-01 rewrites `context.rs` to write the three `.agent/workflows/` files; Plan 12-02 extends `init.rs` with settings.json merge logic. Both plans can be validated with `cargo test` using the existing binary invocation pattern from `test_integration.rs`.

### Pattern 1: Writing .agent/workflows/ files (AGNT-04, AGNT-05, AGNT-06)

**What:** After fetching agents from DB, create the `.agent/workflows/` directory and write three Markdown files. Directory creation is idempotent via `create_dir_all`. Each file is written atomically with `std::fs::write`.

**When to use:** On every `squad-station context` invocation. Files are always overwritten (no merge needed — they are generated from DB state).

**Example:**

```rust
// In context.rs run()
let workflows_dir = std::path::Path::new(".agent/workflows");
std::fs::create_dir_all(workflows_dir)?;

// squad-delegate.md
let delegate_content = build_delegate_md(&agents, &config);
std::fs::write(workflows_dir.join("squad-delegate.md"), delegate_content)?;

// squad-monitor.md
let monitor_content = build_monitor_md();
std::fs::write(workflows_dir.join("squad-monitor.md"), monitor_content)?;

// squad-roster.md
let roster_content = build_roster_md(&agents);
std::fs::write(workflows_dir.join("squad-roster.md"), roster_content)?;

println!("Generated .agent/workflows/ (3 files)");
```

### Pattern 2: settings.json merge with backup (HOOK-03)

**What:** In `init.rs`, after the existing agent launch flow, detect `.claude/settings.json` and/or `.gemini/settings.json`. If a file exists: backup to `.bak`, parse JSON, merge the hook entry under `hooks.Stop` (Claude) or `hooks.AfterAgent` (Gemini), write back. If the file does not exist: print instructions.

**When to use:** On every `squad-station init` invocation. Backup is always written before any modification. The merge is additive — if the hook command is already present, it is not duplicated.

**Example:**

```rust
// In init.rs, after the agent launch flow:
let providers = [
    (".claude/settings.json", "Stop"),
    (".gemini/settings.json", "AfterAgent"),
];
for (settings_path, hook_event) in &providers {
    let path = std::path::Path::new(settings_path);
    if path.exists() {
        merge_hook_entry(path, hook_event)?;
    } else {
        print_hook_instructions(settings_path, hook_event);
    }
}

fn merge_hook_entry(path: &std::path::Path, event: &str) -> anyhow::Result<()> {
    // 1. Backup
    let bak = path.with_extension("json.bak");
    std::fs::copy(path, &bak)?;

    // 2. Parse
    let content = std::fs::read_to_string(path)?;
    let mut value: serde_json::Value = serde_json::from_str(&content)?;

    // 3. Ensure hooks.{event} array exists, then append entry if not already present
    let hook_command = "squad-station signal $TMUX_PANE";
    let entry = serde_json::json!({
        "type": "command",
        "command": hook_command,
    });
    let hooks_array = value
        .pointer_mut(&format!("/hooks/{}", event))
        .and_then(|v| v.as_array_mut());
    // ... insert if not present logic ...

    // 4. Write back
    let output = serde_json::to_string_pretty(&value)?;
    std::fs::write(path, output)?;
    println!("  Updated {} (backup: {})", path.display(), bak.display());
    Ok(())
}
```

### Pattern 3: settings.json structure (confirmed from real project files)

The `.claude/settings.json` format observed in this project:

```json
{
  "hooks": {
    "SessionStart": [ { "hooks": [ { "type": "command", "command": "..." } ] } ],
    "PostToolUse": [ { "hooks": [ { "type": "command", "command": "..." } ] } ]
  },
  "statusLine": { "type": "command", "command": "..." }
}
```

The hook entry for squad-station in Claude Code uses the `Stop` event:

```json
{
  "hooks": {
    "Stop": [
      { "type": "command", "command": "squad-station signal $TMUX_PANE" }
    ]
  }
}
```

Note: Claude Code's `hooks.Stop` array format differs from `hooks.SessionStart` — `SessionStart` entries have a nested `hooks` key, but `Stop` entries are flat `{ type, command }` objects. Use the flat format for `Stop` entries. Confirmed in `docs/SOLUTION-DESIGN.md` section 6.1 code snippet.

The `.gemini/settings.json` format observed:

```json
{
  "hooks": {
    "AfterTool": [ { "hooks": [ { "type": "command", "command": "..." } ] } ]
  }
}
```

For Gemini CLI, the hook event is `AfterAgent`.

### Anti-Patterns to Avoid

- **Writing workflows to stdout instead of files:** The current `context.rs` writes to stdout — this is the exact behavior being replaced. New behavior MUST write to `.agent/workflows/*.md` files.
- **Overwriting settings.json without backup:** HOOK-03 explicitly requires `.bak` backup. Never skip the backup step even if the merge seems trivial.
- **Duplicating hook entries:** Before appending to the hooks array, check if the `"squad-station signal $TMUX_PANE"` command is already present. Do not add duplicates.
- **Hardcoding absolute paths for .agent/workflows/:** Use a relative path `".agent/workflows"` resolved against the current working directory — the user runs `squad-station context` from their project root.
- **Failing init entirely if settings.json is malformed JSON:** Use `serde_json::from_str` with graceful error handling — print a warning and fall through to the "print instructions" branch rather than aborting init.
- **Generating delegate.md with stale agent data:** `context.rs` must re-fetch agents from DB at call time, not cache. Each `context` invocation regenerates from the live DB state.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON merge | Custom string manipulation | `serde_json::Value` deserialization + mutation | Handles all JSON edge cases (whitespace, nested keys, null values) |
| Duplicate-check in hooks array | Regex search | JSON array iteration comparing `.command` field | Type-safe, handles reordering |
| Directory creation | Manual mkdir chain | `std::fs::create_dir_all` | Idempotent, handles nested paths, returns error on permission failure |
| File backup | Custom copy | `std::fs::copy(src, dst)` | Atomic on most OSes, preserves permissions |

**Key insight:** `serde_json` is already in `Cargo.toml`. It handles the entire settings.json lifecycle: parse → mutate `Value` tree → serialize back. No custom JSON parser needed.

## Common Pitfalls

### Pitfall 1: Claude Code Stop hook entry format

**What goes wrong:** The developer uses the `SessionStart`-style nested format `{ "hooks": [{ "type": "command", "command": "..." }] }` for `Stop` entries, which is invalid for that event type.

**Why it happens:** The `.claude/settings.json` in this project uses the nested format for `SessionStart` and `PostToolUse`, leading to confusion. But `Stop` entries in Claude Code use flat `{ "type": "command", "command": "..." }` objects directly in the array.

**How to avoid:** Use the flat format for `Stop` hooks as shown in `docs/SOLUTION-DESIGN.md` section 6.1: `{ "type": "command", "command": "squad-station signal $TMUX_PANE" }`. Test by running a Claude Code session after merge and verifying the hook fires.

**Warning signs:** Hook does not fire after `squad-station init` completes successfully.

### Pitfall 2: Settings.json path — project-relative vs home-relative

**What goes wrong:** Code looks for `~/.claude/settings.json` (global) instead of `.claude/settings.json` (project-local, in the current working directory).

**Why it happens:** Claude Code has both global (`~/.claude/settings.json`) and project-local (`.claude/settings.json`) settings. The design spec (SOLUTION-DESIGN.md §6.1) and existing hook scripts reference the project-local path. The `squad-station init` command runs from the project root, so the path is always relative.

**How to avoid:** Use `std::path::Path::new(".claude/settings.json")` — relative to cwd. Do not expand `~`. Document clearly in the HOOK-04 stdout instructions which file to edit.

**Warning signs:** Init succeeds but no `.claude/settings.json` is modified; user sees the "print instructions" branch even though a settings file exists.

### Pitfall 3: Existing context test breaks after context.rs rewrite

**What goes wrong:** `test_context_lists_registered_agents` in `test_integration.rs` asserts stdout contains agent names. After the rewrite, `context.rs` writes to files and only prints a success summary to stdout — the test fails.

**Why it happens:** The existing test was written for the old stdout-only behavior of `context.rs`.

**How to avoid:** Update the existing test to: (a) assert the 3 files exist in `.agent/workflows/`, (b) assert file contents contain expected agent names. Remove or update the stdout content assertions. The test must run `context` with `current_dir(tmp.path())` so the `.agent/workflows/` directory is created under the temp directory.

**Warning signs:** `cargo test test_context` fails after 12-01 is implemented.

### Pitfall 4: Gemini CLI hook event name

**What goes wrong:** Using `"Stop"` for Gemini CLI (which is Claude Code's event name) instead of `"AfterAgent"`.

**Why it happens:** Both providers are being handled together; it is easy to reuse the Claude Code event name.

**How to avoid:** Use `"AfterAgent"` for `.gemini/settings.json`, `"Stop"` for `.claude/settings.json`. These are provider-specific event names. From `docs/SOLUTION-DESIGN.md` table in section 6.2: Claude Code uses `Stop`; Gemini CLI uses `AfterAgent`.

### Pitfall 5: merge_hook_entry runs even when settings.json is Gemini format

**What goes wrong:** The `.gemini/settings.json` file uses `"AfterTool"` entries with a nested `hooks` key (`{ "hooks": [{ ... }] }`). Merging a flat `Stop`-style entry into a file that uses the nested format creates an inconsistent JSON structure.

**Why it happens:** The two providers have different hook entry schemas. Writing generic merge code treats them identically.

**How to avoid:** The merge function must be parameterized by both (1) event key name and (2) entry format (flat vs nested). Check the existing structure of the hooks array before appending — if entries use the nested `{ "hooks": [...] }` format, match that format for the new entry.

### Pitfall 6: context.rs still references tmux for status reconciliation

**What goes wrong:** The current `context.rs` calls `tmux::session_exists` to reconcile agent status before generating output. For IDE/antigravity setups, no tmux sessions exist for the orchestrator — this reconciliation will incorrectly mark the orchestrator as `dead`.

**Why it happens:** The current `context.rs` was written before the antigravity provider existed. Status reconciliation made sense for all-tmux squads.

**How to avoid:** In the rewritten `context.rs`, skip status reconciliation for agents where `tool == "antigravity"` (or for agents with `role == "orchestrator"` when the squad uses `is_db_only()`). Alternatively, remove reconciliation from `context.rs` entirely — `agents.rs` command already does reconciliation with `squad-station agents`. The `context` command should be read-only with respect to DB state.

### serde_json merge pattern

```rust
// Source: standard serde_json Value manipulation
let content = std::fs::read_to_string(path)?;
let mut root: serde_json::Value = serde_json::from_str(&content)
    .unwrap_or_else(|_| serde_json::json!({}));  // graceful fallback

// Ensure hooks object exists
if root.get("hooks").is_none() {
    root["hooks"] = serde_json::json!({});
}

// Ensure Stop array exists
let stop_arr = root["hooks"]["Stop"]
    .as_array_mut()
    .get_or_insert(&mut vec![]);

let hook_cmd = "squad-station signal $TMUX_PANE";
let already_present = stop_arr.iter().any(|entry| {
    entry.get("command").and_then(|c| c.as_str()) == Some(hook_cmd)
});
if !already_present {
    stop_arr.push(serde_json::json!({ "type": "command", "command": hook_cmd }));
}

let output = serde_json::to_string_pretty(&root)?;
std::fs::write(path, output + "\n")?;
```

## Open Questions

1. **Should `context` still output anything to stdout after writing files?**
   - What we know: The command currently produces all output on stdout. After rewrite, it writes to files.
   - What's unclear: Whether a brief stdout summary (`"Generated .agent/workflows/ (3 files)"`) is desirable for scripting or CI.
   - Recommendation: Print a brief confirmation line to stdout (not to stderr) so users see confirmation. Keep it to 1-2 lines.

2. **Should `init` also call `context` to generate workflow files automatically?**
   - What we know: `docs/SOLUTION-DESIGN.md` section 2 implies init generates context for IDE orchestrators. Phase 12 only explicitly requires it via `context.rs`. The requirements (AGNT-04/05/06) reference only `context.rs`.
   - What's unclear: Whether `init` should auto-call `context` at the end for IDE orchestrators.
   - Recommendation: Keep them separate per requirements. Users run `context` explicitly. This avoids init doing too much and keeps init/context independently invokable.

3. **What happens to the existing `test_context_lists_registered_agents` test?**
   - What we know: It asserts stdout content. After rewrite, stdout is a brief summary only.
   - What's unclear: Whether to update in-place or delete and replace.
   - Recommendation: Replace the existing test with file-content assertions. Check that `.agent/workflows/squad-roster.md` contains agent names. Add separate tests for delegate.md and monitor.md existence.

4. **How to handle `.gemini/settings.json` hook entry format discrepancy?**
   - What we know: `.gemini/settings.json` in this project uses the nested `{ "hooks": [...] }` format for `AfterTool`. The Gemini `AfterAgent` event may use a different format.
   - What's unclear: Whether `AfterAgent` in Gemini CLI uses flat or nested format.
   - Recommendation: Use the flat format matching the design doc. If Gemini's `AfterAgent` uses nested format in practice, the merge code must be updated. Flag this as requiring manual verification against Gemini CLI docs during implementation.
